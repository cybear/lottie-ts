# NEW — Target Architecture

Last updated: 2026-02-19

---

## Goals

1. **Safety** — TypeScript throughout the source; types generated from source, not
   hand-maintained
2. **Clarity** — clean ES modules, no IIFEs, no `var`, no string-substitution tricks
3. **Confidence** — automated test suite that runs on every build
4. **Hygiene** — NPM-only dependency management; no vendored third-party code; no
   build artefacts committed to git
5. **DX** — modern tooling with fast feedback loops

---

produced on CI and published to NPM — not committed to source control.

## Language & type system

| Concern | Target |
|---|---|
| Source language | **TypeScript 5.x** |
| Strict mode | `"strict": true` in `tsconfig.json` |
| Declaration files | Generated by `tsc` — no hand-written `index.d.ts` |
| Internal interfaces | Defined as TypeScript `interface` / `type` in relevant modules |
| `var` keyword | Banned via ESLint `no-var: error` |
| IIFEs for encapsulation | Replaced by module-level `export` / module private semantics |
| Third-party types | Via `@types/*` packages where available |

Example of what the AnimationManager would look like:

```ts
// src/animation/AnimationManager.ts
import type { AnimationItem } from './AnimationItem';

interface RegisteredAnimation {
  elem: Element | null;
  animation: AnimationItem;
}

const registeredAnimations: RegisteredAnimation[] = [];
// ...
```

## Module system

- Source: pure **ES modules** (`.ts` files, `import`/`export`)
- No global variable mutation in source files
- The UMD bundle's global attachment (`window.lottie`) is handled entirely by the
  Rollup output configuration, not by source code
- Standalone-mode embedding is handled through a separate, explicit build target
  (not by string-replacing source literals)

## Build system

### Primary tool: Rollup 4.x (or tsup as a simpler alternative)

Rollup remains a good fit because the library ships multiple entry points
(svg-only, canvas-only, full, light variants) and fine-grained tree-shaking
control is needed.

Key changes from the current setup:

| Current | Target |
|---|---|
| `@rollup/plugin-babel` + `@babel/preset-env` | `@rollup/plugin-typescript` (via `tsc`) |
| Babel transpilation to ES5 | TypeScript target `ES2020`; consumers own polyfills |
| `rollup-plugin-terser` | `@rollup/plugin-terser` (maintained fork) |
| Custom string-replace plugins | Eliminated; version injected via `import pkg from './package.json' assert { type: 'json' }` |
| Navigator/document guard as string prepend | Proper conditional export or explicit guard block |
| `tasks/build_worker.js` as post-process | Worker entry point treated as a first-class Rollup input |
| Legacy `tasks/build.js` | Deleted |

### Output formats

| Format | File | Purpose |
|---|---|---|
| ESM | `dist/esm/*.js` + `.d.ts` | Tree-shakeable; recommend for bundlers |
| CJS | `dist/cjs/*.js` + `.d.ts` | Node.js `require()` compatibility |
| UMD | `dist/umd/lottie.js` / `lottie.min.js` | CDN / bare `<script>` tag |

`package.json` exports field:

```json
{
  "exports": {
    ".": {
      "import": "./dist/esm/index.js",
      "require": "./dist/cjs/index.js",
      "types": "./dist/esm/index.d.ts"
    },
    "./svg": { "import": "./dist/esm/svg.js", ... },
    "./canvas": { "import": "./dist/esm/canvas.js", ... }
  },
  "module": "./dist/esm/index.js",
  "main": "./dist/cjs/index.js",
  "types": "./dist/esm/index.d.ts"
}
```

## Testing

### Unit tests — Vitest

```
src/
  utils/
    bez.ts
    bez.test.ts       ← co-located unit test
```

- **Vitest** (runs in Node.js via `jsdom` environment for DOM-dependent code)
- Tests import directly from TypeScript source — no pre-build needed
- Coverage with `@vitest/coverage-v8`
- Run with `npm test`

### Integration / visual tests — Puppeteer

- Kept in `e2e/` directory
- Uses the same Puppeteer + pixelmatch approach as the current `tests/verification/`
- Run against the built `dist/` output: `npm run build && npm run test:e2e`
- Reference screenshots stored in `e2e/snapshots/` (committed to git)

### CI script (example)

```
npm ci
npm run lint
npm run build
npm test          # Vitest
npm run test:e2e  # Puppeteer visual tests
```

## What is explicitly NOT changing

- The **public API** of lottie itself (the `loadAnimation`, `play`, `stop`, etc.
  interface) is not being redesigned — this is a tooling/language migration, not an
  API rewrite
- The **rendering logic** is preserved as-is; files are migrated incrementally
- UMD global output (`window.lottie`) is kept for CDN users

---

## Non-goals

- Converting the project to a monorepo (not needed at current scale)
- Switching rendering engines or supporting new animation formats
- Rewriting expression evaluation in a sandboxed runtime
